# karate-announce 開発ルール

> **⚠️ このファイルは必ず最後まで全行を読むこと。途中で読むのをやめない。**

> **このファイルが唯一のルール源（Single Source of Truth）**
> 「覚えて」「次からやって」等の継続的指示は必ずこのファイルに追記する。
> Memory ファイルは補足情報（背景・外部リファレンス）のみ。ルール自体は置かない。
> この方針はユーザーと合意済み（2026-03-25）。理由：Memory ファイルはコンテキスト圧縮で
> 失われることがあり、CLAUDE.md だけが毎セッション確実に読み込まれるため。

---

## プロジェクト概要

- Next.js 16（App Router）+ TypeScript + Tailwind CSS 4
- DB: Supabase（PostgreSQL）
- 音声: OpenAI TTS
- デプロイ: Vercel（Stop hook 経由で自動）
- パッケージマネージャ: npm
- ブランチ戦略: **main 直接プッシュ**（feature ブランチは使わない）
- Lint: ESLint（eslint-config-next + @typescript-eslint + import）。`npm run lint` で実行。警告 0 を維持すること
- Formatter: Prettier。`npm run format` で実行。`npm run format:check` で検証
- **eslint-disable コメントは絶対に使用禁止。** 例外なし。pre-commit hook で検出しブロックする。lint 警告は根本原因を修正すること

## 環境変数

- `.env.local` で管理。`.gitignore` 済み。
- 新しい環境変数を追加する場合は、Vercel の Environment Variables にも追加すること。
- 必要な変数: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `OPENAI_API_KEY`, `ADMIN_USERNAME`, `ADMIN_PASSWORD`, `RESEND_API_KEY`, `RESEND_FROM_EMAIL`, `NEXT_PUBLIC_APP_MODE`

## Vercel デプロイ時の注意事項

- **レート制限**: 公開エントリーAPIのレート制限はin-memory Map実装。Vercelのサーバーレス関数はコールドスタートでメモリがリセットされるため、トラフィック集中時にレート制限が効かない場合がある。大会当日に大量申込が予想される場合は、Supabase RLS や外部レート制限サービス（Upstash等）への移行を検討すること。
- **関数タイムアウト**: Vercel Free は10秒、Pro は60秒。メール送信を含むエントリーAPIは処理時間が長くなる可能性がある。

## 計画（大きな機能の場合）

- 複数ファイルにまたがる変更や新機能追加の場合は、**先に計画を出してレビューを受けてから実装する**。
- 小さなバグ修正や1ファイルの変更は直接実装してよい。
- **計画→実装のフロー:**
  - 計画が承認されたら、**基本はメインセッションで直接実装する**（文脈を活かせて精度が高い）
  - 特に大きなタスク（多ファイル変更、長時間が予想される場合）のみ、implementer での実行を提案する：「長時間かかるおそれがあるので implementer での実行をおすすめします。どうしますか？」
  - implementer を使う場合は `/tmp/claude-plan.md` に計画を書き出し、`/tmp/claude-implementer-running` でフラグ管理。push先はプロジェクト設定を引き継ぐこと

## 実装完了フロー（TDD 駆動・この順序で必ず実行）

> **重要**: このフローは機能追加・修正・バグ修正すべてに適用する。順序をスキップしない。
> ユーザーが「開発フローに従って」「ルールに従って」「修正して」等、どのような表現で指示しても、このフローを必ず守る。明示的な指示がなくても、コード変更を伴う作業には常にこのフローを適用する。

**実装開始時のタスク作成（必須）**: 実装に入る前に TaskCreate で Step 1〜7 のタスクリストを作成する。タスクリストがチェックリストとして機能し、ステップの飛ばしを防ぐ。各 Step の作業開始時に in_progress、完了時に completed にする。

**Agent ツールの使用ルール（実装作業）**:

- **Step 1〜7の実装作業（コード編集・コミット・プッシュ）ではAgent ツール禁止。メインセッションで直接実行する。** エージェントはコンテキストを持ち直すオーバーヘッドがあり、確認待ちで停止するリスクもあるため、メインセッションの方が速く確実。
  - **例外**: ユーザーが明示的に「並列で」「エージェント使って」と指示した場合のみ使用可。
- **調査・レビュー（コード読み取りのみ）ではAgent（Explore等）使用可。** 複数の観点で並列レビューする等の用途。

### Step 1: 影響範囲の把握 → 仕様書を先に更新

- **実装の前に、変更対象のファイルで `scripts/impact-check.sh` を実行し、関連する仕様書・テスト・画面を把握する。** 列挙されたファイルを読み、現状を理解してから仕様書の更新に入る。これにより「仕様書にあるのに未実装」「テストがあるべきなのにない」等の既存の不整合にも気づける。
- **実装の前に仕様書を更新する。** コード変更より先に「何を作るか」を仕様書に記載する。
- 新機能 → `docs/` に詳細仕様を追記。SPEC.md に画面/APIが増える場合は一覧にも追加。
- 既存機能の変更 → `docs/` 内の該当仕様書の記述を修正。
- バグ修正 → 仕様書の記述が正しいか確認し、必要なら仕様書も修正。
- **SPEC.md と docs/ の役割分担:**
  - `SPEC.md`: 全体概要のみ。画面一覧は名前+1行説明+docs/へのリンク。機能詳細は書かない
  - `docs/`: 各機能の詳細仕様。新機能の詳細はここに書く
- **SPEC.md 変更履歴**: 種類（新機能・変更・バグ修正）を問わず、**全ての変更で必ず1行追記する**（Step 5 で SPEC.md が diff に含まれる必要があるため）。
  - **書式**: `- **機能名（YYYY-MM-DD）**: 1行説明`（例: `- **よみがな必須の親フィールド連動（2026-04-03）**: 親が任意なら読み仮名も自動的に任意にする`）
  - **肥大化防止**: 変更履歴の行数が100行を超えたら、古い方から `docs/CHANGELOG.md` に移動して50行まで減らす。`docs/CHANGELOG.md` には日付の新しい順で追記する。

### Step 2: テストを先に書く（Red）

- **仕様書の記述に基づいてテストを追加・修正する。** 実装の前にテストを書く。
  - API ルートの追加・変更 → `__tests__/api/` に正常系テストを追加・更新
  - `lib/` の純粋ロジック追加・変更 → `__tests__/unit/` にユニットテストを追加・更新
  - UIの追加・変更・リファクタ時は関連する E2E テスト（`__tests__/e2e/`）も追加・更新する
  - **E2E テスト（現在無効化中）**: E2E テストは**書くが実行しない**。本番リリース前に `npx playwright test` で全通過を確認すること。E2Eは本番DBに接続して実行されるため、既存データを絶対に削除・変更しない。テストデータは自分で作成し、終了時に削除する。
- **ユニット/API テストで検証困難な場合**（コンポーネント内のレンダリングロジック等）: E2E テストで代替するか、手動確認で代替する。その場合はコミットメッセージに「手動確認済み: 〇〇」と記載する。
- テストを実行し、**追加したテストが失敗することを確認する**（Red 状態）。
  - 失敗しない場合 → テストが正しく書けていないか、既に実装済み。テストを見直す。
- 既存テストが壊れた場合はテストを修正してから進める（テストを削除して通すのは禁止）。
- E2E テストが壊れた場合は、まず実装とテストのどちらが正しいか調査・判断する。意図的なUI変更ならテストを更新、意図しない変更（リグレッション）なら実装側を修正する。
- **テストファイル命名規約**:
  - ユニット / API テスト: `*.test.ts`（例: `bracket.test.ts`）
  - E2E テスト: `*.spec.ts`（例: `entry-form.spec.ts`）

### Step 3: 実装 → 全チェック通過

- `npm run dev`, `curl` 等は確認不要で実行してよい。
- テストが通るように実装する。
- 以下を**すべて**実行し、全パスを確認する（pre-commit hook で弾かれる手戻りを防ぐため、hook と同等以上のチェックをここで実施する）:
  1. `npm run format:check` — Prettier フォーマット検証。失敗時は `npm run format` で修正し差分を確認
  2. `npx eslint . --max-warnings 0` — ESLint（警告も0件必須）
  3. `npx tsc --noEmit` — 型チェック
  4. `npx vitest run` — 全テスト通過（Green 状態）
  5. `npm run build` — ビルドエラーなし（**ページ・レイアウト・config・CSS の変更を含む場合は必須**。API/lib のみの変更時は省略可。CI + Vercel が安全網）
  6. `npm ci` — lock file 整合性検証（**package.json を変更した場合のみ**）
- いずれかが失敗した場合は、この Step 3 内で修正→再実行を繰り返す。Step 4 以降には進まない。

### Step 4: レビュー

> pre-commit hook が自動チェック済みの項目（RPC/マイグレーション整合性、SPEC.md日付・参照・API一覧、認証チェック、ESLint、型チェック、テスト実行、gitleaks、semgrep）は手動確認不要。以下は hook では検出できない項目のみ。

**4a. 影響範囲の確認**

`scripts/impact-check.sh` を実行し、列挙された全ファイル（仕様書・テスト・画面・コンポーネント）を読む。

**4b. 整合性チェック**

以下の3点が完全に一致するまで修正を繰り返す:

1. **仕様書の記述**（docs/ + SPEC.md）— 今回の変更と矛盾する記述がないか。仕様書に記載されているが未実装の機能はないか
2. **テストケース** — 今回の変更に対するテストが不足していないか。テストの期待値が仕様に沿っているか
3. **実装コード** — 今回の変更が正しく統合されているか

**テストを削除・移行する場合**: 削除する各テストケースの観点を列挙し、残存テストでカバーされているかを1件ずつ対応表にして確認する。カバーされていない観点があれば、残存テストにケースを追加してからでなければ削除しない。対応表はユーザーに提示して確認を取る。

**4c. チェックリスト（該当する場合のみ）**

- **データフロー検証（APIを変更・追加した場合）**: フロント送信フィールド名 ↔ API受取フィールド名 ↔ DB保存フィールド名の3点一致
- **エラーハンドリング（fetch を追加・変更した場合）**: `res.ok` チェック、エラー時のUI状態、ユーザー通知
- **ローディング状態（ボタン・フォームを追加・変更した場合）**: disabled 制御、ローディング中の空表示防止
- **表示確認（データの保存・取得を変更した場合）**: 保存→反映→リロードの一連確認
- **ナビゲーション（ページを追加・移動した場合）**: リンク追加漏れ、URL直アクセスのみのページ防止
- **UX ポリシー（ページを追加・変更した場合）**: 5項目（ローディング・エラー区別・テーマ・オフライン・a11y）
- **状態管理（state・ref・イベントリスナー・永続化を追加・変更した場合）**: 状態遷移全パターン、リソース取得/解放の対、永続/揮発データ整合性
- **エクスポート関数（lib/ にモジュールを追加・変更した場合）**: 全呼び出し元確認、dead export 検出、統合漏れ確認。**変更した lib/ ファイルの全 export について、アプリコード（app/ components/）から import されているかを grep で確認する。テストのみから import されている export は、export を外して間接テストに移行するか、アプリコードで使用する。** pre-commit hook がブロックするため、ここで検出しないと Step 5 で差し戻しになる
- **マルチテナント（DBテーブル・設定・認証を変更した場合）**: ハードコード排除、環境変数切替、tenant_id 分離可能性

**4d. 壊れ方の検証（全項目に回答必須。該当なしも明記。完了時にユーザーに一覧を提示する）**

- **異常系パス**: 変更した関数内の全 error/catch/throw パスを追跡。中間データが残存しないか。エラーレスポンスに DB 内部情報（テーブル名・カラム名・SQL）が漏洩しないか
- **波及（2層で確認）**:
  - **参照元**: 変更した関数・型・カラムの全参照元を grep で列挙し、呼び出し元が新しいインターフェースに対応しているか確認
  - **データパターン**: 新しいデータパターン（フラグ・ステータス値・選択肢・フィールド等）を追加した場合、そのデータを**生成（テストデータ含む）・表示・フィルタ・ソート・エクスポート（CSV等）・集計**する全コードが新パターンに対応しているか。grep でデータ構造名・フィールド名を検索して網羅的に確認する
- **N+1・実行頻度**: ループ内 DB クエリがないか。useEffect の deps にオブジェクト/配列リテラルを入れていないか。ポーリング・購読の重複がないか
- **境界値**: 空配列・null・undefined・空文字・存在しないID・型違い・大量データでコードパスを追跡
- **部分失敗**: 複数テーブル操作で途中失敗時、クリーンアップが今回追加したテーブルも含め全対象になっているか
- **順序依存**: 関数の呼び出し順を入れ替えたら壊れる箇所はないか。ある場合はコメントで明記されているか

**4e. UI 変更時はブラウザで目視確認**

- **デプロイ後に本番サイト（https://karate.naocho.net/）で確認する。** localhost での確認は認証・環境差異で無駄な問題が発生するため行わない
- 要素の重なり・はみ出し・余白の崩れがないか目視チェック
- コードの diff だけで「問題ない」と判断しない

**問題を見つけたら修正 → Step 3 の全チェックを再実行してから 4a に戻る。**

### Step 5: コミット

- **1つの実装に対して1回。分割コミット禁止。**
- `git status` で意図しないファイル（デバッグ用一時ファイル等）が含まれていないか確認してから `git add -A && git commit` を1回だけ実行する。
- SPEC.md が diff に含まれていないコミットは禁止。SPEC.md のヘッダー「最終更新」日付も当日に更新すること。
- コミットメッセージは実装内容を日本語で簡潔に（例：「対戦表作成後に該当トーナメントへスクロール」）。
- 未コミットの変更が残っていると Stop hook がブロックする。必ず commit まで完了させてから応答を終えること。

### Step 6: プッシュ

- コミット後に `git push origin main` を実行する。（Vercel の GitHub 連携が自動デプロイする）
- `vercel --prod` は実行しない。

### Step 7: デプロイ完了確認

- 完了報告の前に `vercel ls --scope naocho-fs-projects` でデプロイが `● Ready` になっていることを確認する。
- Ready になっていなければ30秒待ってから確認、以降10秒ごとにリトライして Ready を待つ。
- **デプロイ完了を確認してから報告する。**

## DBスキーマ変更

- DBカラム追加・テーブル作成等が必要な場合、**ユーザーに依頼せず自分で** Supabase Management API 経由で実行する。確認も不要。
- 実行方法は memory/reference_supabase_sql.md を参照。
- 既存データのマイグレーションも同時に実行する。
- **マイグレーションファイル必須**: SQL を Supabase に実行したら、同じ内容を `supabase/migrations/` にも `CREATE OR REPLACE` 形式で保存する。RPC 関数・テーブル変更・インデックス等すべて対象。マイグレーションファイルがないと新環境構築時に再現できない。
- **`supabase_schema.sql` も同時更新必須**: カラム追加・テーブル作成時は `supabase_schema.sql` にも反映する。`lib/types.ts` の型定義と `supabase_schema.sql` のカラム定義が一致していることを確認する。

## コミュニケーション

- 日本語は丁寧語（「〜しました」「〜です」）を使うこと。タメ口・体言止め禁止。
- 確認は最小限にする。このファイルに手順が書いてあることは確認不要で実行する。
- **ユーザーから入力があったら、必ず返事（了解・質問への回答等）をしてから作業に入る。** 返事なしで作業を開始することは禁止。ツール呼び出しより先にテキストで応答すること。

## 判断ルール

- **機能の削除・無効化・方針変更を勝手に判断しない。** 必ずユーザーに確認してから行う。
- 「〜が難しいので諦めます」ではなく「〜が難しいですが、A案/B案どちらにしますか？」と選択肢を提示する。
- 判断に迷うことは全てユーザーに聞く。自分の判断で進めない。
- **最もシンプルな解決策を最初に選ぶ。** ベストプラクティスが明確なものは迷わずそれを選ぶ。回りくどいワークアラウンドを検討する前に、正攻法で解決できないか考える。

## ミス防止

- **作業中に不要な確認を挟まない。** 「続けますか？」「どうしますか？」等の確認で作業を止めない。作業完了まで手を動かし続ける。セッション切り替えが必要な場合は「切り替えてください」と断定的に言う（「〜しますか？」と聞かない）。判断に迷う場合のみ質問する。
- **セッション切り替え時は引き継ぎを準備する。** 切り替えを提案する前に以下を実行する:
  1. `/tmp/` に引き継ぎファイルを作成（残タスク・状態・修正パターン）
  2. Memory ファイルを最新状態に更新
  3. 次のセッションで入力するプロンプト例を提示する
- 同じミスを2回繰り返したら、3回目が起きないよう仕組み（Hook・スクリプト・設定）で対処する。
- 「次から気をつけます」だけで終わらせない。対策後は何をどう変えたか報告する。
- **自分で宣言したことは必ず実行する。** 「今後は〜します」と言ったらそれはルールになる。守れないなら言わない。
- **同じ修正を2回試しても改善しない場合**: 立ち止まって根本原因を分析する。3回目の同じアプローチは禁止。代わりに以下を行う:
  1. 問題を再定義する（本当の原因は何か？）
  2. 関連コードを広く読み直す（狭い範囲だけ見ていないか？）
  3. 異なるアプローチを検討してユーザーに提示する
- **フィールド追加・変更時は影響範囲を網羅的に確認する。** DB（カラム）→ 型定義 → API（受け取り・保存）→ UI（入力・表示）→ ロジック（フィルタ・計算）→ テスト、の全レイヤーを漏れなく修正する。1箇所でも漏れるとデータが保存されない・表示されない不具合になる。
- **あらゆる追記・変更時の確認**: CLAUDE.md・settings.json・Memory 等、どのファイルへの追記・変更でも、既存内容との重複・矛盾・競合がないか毎回確認すること。
- **レビューで見つけた問題は全て修正してからコミットする。** 「後で」「影響軽微」で先送りしない。問題を認識しながら放置するのは隠蔽と同じ。
- **問題・警告・懸念は全てユーザーに報告する。** 自分で「影響なし」「軽微」と判断して報告を省略しない。ユーザーに不利益が生じる可能性がある情報を握りつぶさない。判断はユーザーがする。
- **明らかな自分のミスは `claude_mistakes` テーブルに記録する。** CLAUDE.md のルールを無視した、同じミスを繰り返した等、自分に非がある場合は Supabase の `claude_mistakes` テーブルに INSERT する。カラム: `mistake`（何をやらかしたか）, `user_instruction`（ユーザーの指示）, `my_response`（自分がどう応答したか）, `what_went_wrong`（なぜ間違えたか）, `what_should_have_done`（正しい行動）, `prevention_taken`（再発防止策）, `category`（分類）, `severity`（high/medium/low）。

## 品質チェックの役割分担

3層で役割を分けている。同じチェックを複数層で重複させない。
| 層 | タイミング | 担当 | 所要時間 |
|---|----------|------|---------|
| **pre-commit hook** | コミット前（ローカル） | 静的チェック + ESLint(staged) + tsc + vitest + gitleaks(staged) + semgrep(staged) | ~15秒 |
| **GitHub Actions CI (test.yml)** | プッシュ後（GitHub） | 静的チェック + ESLint + tsc + vitest + build | ~70秒 |
| **GitHub Actions CI (security.yml)** | プッシュ後（GitHub） | ESLint + Semgrep + osv-scanner + gitleaks + npm audit + CodeQL | ~5分 |
| **Vercel** | デプロイ時 | build | 自動 |

- pre-commit は高速であること最優先。build は CI と Vercel に任せる。
- CI は hook バイパス時の安全網。hook と同じ静的チェックに加え build も実行。
- ソース: `scripts/pre-commit`（Git 管理）。`npm install` 時に `prepare` スクリプト（husky）で自動セットアップ。
- `.husky/pre-commit` → `scripts/pre-commit` の呼び出しチェーン。hook 内容の変更は `scripts/pre-commit` のみ編集。
- Semgrep false positive 抑制: `// nosemgrep: <rule-id>` コメントは semgrep 固有の抑制機構であり、eslint-disable とは別ツール。
- 警告のベースライン: `.warnings-baseline`。新規追加分のみ表示し、既知の警告による疲れを防止。

## Dependabot PR の対応

Dependabot PR が作成されたら、ユーザーに確認せず自分で判断して対応する。

- **パッチ/マイナー更新**: マージ（`gh pr merge --merge --admin`）
- **メジャーバージョンアップ**: マージせずユーザーに報告（PR番号・変更内容・破壊的変更の有無・影響範囲）。判断はユーザーが行う
- **react + react-dom**: 必ずセットで更新。片方ずつマージするとバージョン不一致になる
- **コンフリクト発生時**: `@dependabot rebase` コメントで rebase させてからマージ
- ローカルを `git pull` で同期して完了

## UX ポリシー（全ページ必須）

レビューで繰り返し発見された UX 問題をポリシー化。新規ページ作成・既存ページ変更時に必ず確認する。

**1. ローディング状態**

- データ取得中は**必ずスピナーを表示**する。空白画面（`<div className="min-h-screen bg-main-bg" />`）は禁止。
- スピナー: `<div className="w-8 h-8 border-2 border-blue-500 border-t-transparent rounded-full animate-spin" />`

**2. エラー状態の区別**

- 「データなし（正常）」と「取得失敗（エラー）」を**別の UI で表示**する。
- エラー時は原因の説明 + 再読み込みボタンを表示する。
- 例: 「準備中です」（正常）vs「情報の取得に失敗しました。再読み込み」（エラー）

**3. テーマ統一**

- 全ページで `bg-main-bg`（ダーク）を使用。`bg-gray-50` 等のライト背景は禁止。
- テキスト: 白 `text-white`、補助 `text-gray-400`、非活性 `text-gray-500`

**4. オフラインモード**

- API 呼び出しがあるページは**全て**オフラインモードに対応する（`useOfflineMode` + `flush` + 状態表示）。
- 新規ページ追加時は、既存のコート画面（`app/court/[court]/page.tsx`）をパターンとして参照する。

**5. アクセシビリティ**

- フォーム入力要素は `<label htmlFor="id">` と `<input id="id">` で紐付ける。

## 不具合報告の対応フロー

- ユーザーから不具合報告の修正を依頼されたら（「不具合修正お願い」「バグ直して」等）:
  1. `GET /api/bug-reports` で未対応（status: open）の報告を取得
  2. **一覧と方針を提示して承認を得る**:
  - 報告の一覧（件数・各報告の要約）を提示する
  - 各報告について原因の見立てと修正方針を簡潔に説明する
  - 再現できない・内容が不明な報告は「確認が必要」と明示する
  - 複数件ある場合は優先順の提案も含める
  - **ユーザーの承認を得てから修正に着手する**
  3. 承認後、`PATCH /api/bug-reports/{id}` で `{"status":"in_progress"}` に変更し、**TaskCreate で Step 1〜7 のタスクリストを作成してから** TDD 駆動で修正を実施（実装完了フローに従う）
  4. **1件修正するごとに**コミット→ `PATCH /api/bug-reports/{id}` でステータス（resolved）・対応内容・修正バージョンを更新する。関連する複数報告をまとめて1コミットで修正してもよい
  5. 全件完了後、管理画面（設定 → 不具合報告）で確認できる状態にしてから、対応結果をまとめて報告する
- API呼び出し方法: admin_auth クッキーの値は `/tmp/admin-auth-cookie` から読み取る。ファイルが存在しない場合は `.env.local` の `ADMIN_PASSWORD` からハッシュ計算して生成する（ユーザーに確認不要）。

  ```
  # クッキー生成（/tmp/admin-auth-cookie が存在しない場合）
  node -e "const crypto = require('crypto'); const pw = 'ADMIN_PASSWORDの値'; console.log(crypto.createHash('sha256').update(pw + 'karate-announce-v1').digest('hex'));" > /tmp/admin-auth-cookie

  # 取得（結果をファイルに保存してから処理）
  curl -s "https://karate.naocho.net/api/bug-reports" -H "Cookie: admin_auth=<COOKIE値>" -o /tmp/bug-reports.json
  # → 別コマンドで python3 -c "..." で /tmp/bug-reports.json を読む

  # 更新
  curl -s "https://karate.naocho.net/api/bug-reports/{id}" -X PATCH -H "Cookie: admin_auth=<COOKIE値>" -H "Content-Type: application/json" -d '{"status":"resolved","resolution":"対応内容","fixed_in_version":"コミットSHA先頭7桁"}'
  ```

## 全体レビュー

- 「全体レビュー」「全体レビューして」と指示されたら、以下の全観点でレビューを実行する:
  1. **docs/ 仕様書と実装の一致**: docs/ 内の各仕様書に記載された仕様が、実際のコードと一致しているか
  2. **仕様の矛盾・一貫性**: 仕様書内で矛盾した記述や一貫性のない定義がないか
  3. **仕様書間の整合性**: SPEC.md と docs/ 内の個別仕様書の間で齟齬がないか
  4. **テストカバレッジ**: 仕様に記載されているがテストが実装されていない箇所はないか（ユニットテスト・APIテスト・E2Eテスト全て）
  5. **テスト期待結果と仕様の紐づき**: テストの期待値が仕様書の記述と整合しているか
  6. **その他必要な観点**: セキュリティ、エッジケース、型安全性、パフォーマンス等
- pre-commit hook が既に自動チェックしている項目（RPC整合性、SPEC.md日付、仕様書参照、APIテスト存在）はレビューで再確認不要。
- 複数の Explore エージェントを並列で使い効率的に調査する。
- **レビュー指摘の恒久対策**: レビューで問題を発見したら、修正に加えて以下を行う:
  - 機械的に検出可能 → `scripts/pre-commit` にチェックを追加（コミットをブロック or 警告）
  - 振る舞いの正しさ → テストを追加（削除・変更したら落ちる）
  - どちらにも該当しない → 具体的な理由とともにユーザーに報告
