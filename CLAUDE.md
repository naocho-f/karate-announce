# karate-announce 開発ルール

> **このファイルが唯一のルール正（Single Source of Truth）**
> 「覚えて」「次からやって」等の継続的指示は必ずこのファイルに追記する。
> Memory ファイルは補足情報（背景・外部リファレンス）のみ。ルール自体は置かない。
> この方針はユーザーと合意済み（2026-03-25）。理由：Memory ファイルはコンテキスト圧縮で
> 失われることがあり、CLAUDE.md だけが毎セッション確実に読み込まれるため。

---

## プロジェクト概要
- Next.js 16（App Router）+ TypeScript + Tailwind CSS 4
- DB: Supabase（PostgreSQL）
- 音声: OpenAI TTS
- デプロイ: Vercel（Stop hook 経由で自動）
- パッケージマネージャ: npm

## コミュニケーション
- 日本語は丁寧語（「〜しました」「〜です」）を使うこと。タメ口・体言止め禁止。
- 確認は最小限にする。このファイルに手順が書いてあることは確認不要で実行する。

## 計画（大きな機能の場合）
- 複数ファイルにまたがる変更や新機能追加の場合は、**先に計画を出してレビューを受けてから実装する**。
- 小さなバグ修正や1ファイルの変更は直接実装してよい。
- **計画→実装の自動フロー:**
  1. 計画が承認されたら `/tmp/claude-plan.md` に最終計画を書き出す（`.claude/` 内は権限確認が出るため `/tmp/` を使う）
  2. `touch /tmp/claude-implementer-running` でフラグを作成（Stop hook の完了通知を抑制）
  3. `implementer` エージェントを起動して実装を委譲する（フレッシュなコンテキストで実行）
  4. implementer が計画に沿って実装・テスト・コミットまで完了し、`/tmp/claude-plan.md` を削除する
  5. implementer がフラグを削除し、完了通知を鳴らす
  6. 結果サマリーがメインセッションに返る
- ユーザーが「プランして」→ 承認 → 自動で implementer 起動、が基本フロー。
- セッション開始時に `.claude/plan.md` が残っていたら、前セッションの未実装計画として通知される。
- **implementer 実行中の作業制御:**
  - 同じプロジェクトのファイルを編集しない（コミット競合を防ぐ）
  - ユーザーから依頼があった場合は、理由を説明して実装はせず、タスク化（plan.md への追記や次の計画として記録）してスタックする
  - implementer 完了後にスタックした作業を順次実行する
  - 別プロジェクトの作業や相談は可能

## 実装完了フロー（この順序で実行）

### Step 1: テスト
- `npm run dev`, `curl`, `npx tsc --noEmit` 等は確認不要で実行してよい。
- **テストなしの完了報告は禁止。** 必ず動作確認を完了させること。
- **実装・リファクタには必ず自動テストをセットで書く。** テストなしのコード変更はコミットしない。
  - API ルートの追加・変更 → `__tests__/api/` に正常系テストを追加・更新
  - `lib/` の純粋ロジック追加・変更 → `__tests__/unit/` にユニットテストを追加・更新
  - 既存テストが壊れた場合はテストを修正してから進める（テストを削除して通すのは禁止）
- コミット前に `npx vitest run` で全テスト通過を確認する。

### Step 2: チェックリスト（該当する場合のみ）

**データフロー検証（APIを変更・追加した場合）**
- フロントエンドが `fetch` で送っているフィールド名
- APIルートが `request.json()` で受け取っているフィールド名
- `insert` / `update` でDBに渡しているフィールド名
- → 3点が一致しているか確認。不一致があれば修正。

**エラーハンドリング確認（fetch を追加・変更した場合）**
- `fetch` のレスポンスを `res.ok` でチェックしているか
- APIエラー時に UI が誤った状態に更新されないか（楽観的更新は特に注意）
- エラー時にユーザーに通知しているか（alert または UI 表示）

**ローディング状態確認（ボタン・フォームを追加・変更した場合）**
- 処理中はボタンが `disabled` になっているか
- ローディング中に空のリストや「登録なし」テキストが表示されないか

**表示確認（データの保存・取得を変更した場合）**
- 保存したデータが画面に正しく反映されるか、コードを追って確認
- 画面を再ロードしたときに正しく表示されるか

**ナビゲーション確認（ページを追加・移動した場合）**
- 新規ページへのリンク/タブが既存のナビゲーションに追加されているか
- URL直アクセスでしか到達できないページが存在しないか
- 全管理画面ページがメインの管理画面（`/admin`）のタブ・サブタブからアクセス可能か確認

**マルチテナント化への配慮（DBテーブル・設定・認証を変更した場合）**
- 団体固有のハードコード値（テキスト・ルール等）がないか → DB or config に外出しする
- 新しい外部サービス連携は環境変数で切り替え可能か
- Supabase 固有機能に依存していないか（PostgreSQL 標準の範囲で書く）
- 将来 `tenant_id` で分離できる構造になっているか

### Step 3: ビルド確認
- `npm run build` でビルドエラーがないことを確認する。

### Step 4: 仕様書更新（SPEC.md + docs/）
- **指示されなくても自発的に**更新する。毎回指示を待たない。
- `SPEC.md`（全体仕様）と、変更に関連する `docs/` 内の詳細仕様書の両方を更新する。
- コード変更がすべて完了した後、コミット直前に1回だけ行う。
- 途中で更新しない（後でコード変更すると Stop hook のフラグが再セットされる）。

### Step 4.5: セルフレビュー
- コミット前に、変更した箇所が仕様書（SPEC.md + docs/）と整合しているかセルフチェックする。
- 確認項目: 実装が仕様通りか、テストの期待値が仕様に沿っているか、仕様に記載した内容にテストがあるか。
- 不整合を見つけたら修正してからコミットする。

### Step 5: コミット
- **1つの実装に対して1回。分割コミット禁止。**
- `git add -A && git commit` を1回だけ実行する。
- SPEC.md が diff に含まれていないコミットは禁止。
- コミットメッセージは実装内容を日本語で簡潔に（例：「対戦表作成後に該当トーナメントへスクロール」）。
- 未コミットの変更が残っていると Stop hook がブロックする。必ず commit まで完了させてから応答を終えること。

### Step 6: デプロイ（自動）
- `vercel --prod` や `git push` は実行しない。Stop hook が自動で行う。

### Step 7: デプロイ完了確認
- 完了報告の前に `vercel ls` でデプロイが `● Ready` になっていることを確認する。
- Ready になっていなければ5秒ごとにリトライして Ready を待つ。
- **デプロイ完了を確認してから報告する。**

## DBスキーマ変更
- DBカラム追加・テーブル作成等が必要な場合、**ユーザーに依頼せず自分で** Supabase Management API 経由で実行する。確認も不要。
- 実行方法は memory/reference_supabase_sql.md を参照。
- 既存データのマイグレーションも同時に実行する。

## ミス防止
- 同じミスを2回繰り返したら、3回目が起きないよう仕組み（Hook・スクリプト・設定）で対処する。
- 「次から気をつけます」だけで終わらせない。対策後は何をどう変えたか報告する。
- **自分で宣言したことは必ず実行する。** 「今後は〜します」と言ったらそれはルールになる。守れないなら言わない。
- **同じ修正を2回試しても改善しない場合**: 立ち止まって根本原因を分析する。3回目の同じアプローチは禁止。代わりに以下を行う:
  1. 問題を再定義する（本当の原因は何か？）
  2. 関連コードを広く読み直す（狭い範囲だけ見ていないか？）
  3. 異なるアプローチを検討してユーザーに提示する

## あらゆる追記・変更時の確認
- CLAUDE.md・settings.json・Memory 等、どのファイルへの追記・変更でも、既存内容との重複・矛盾・競合がないか毎回確認すること。

## UIスタイルガイド
- **タブのレイアウト**: サブタブは `grid grid-cols-{タブ数}` で横幅を均等割にする。`flex` で左寄せにしない。

## 不具合報告の対応フロー
- ユーザーから「不具合を直して」と依頼されたら:
  1. `GET /api/bug-reports` で未対応（status: open）の報告を取得
  2. **修正開始時に** `PATCH /api/bug-reports/{id}` で `{"status":"in_progress"}` に変更する
  3. 内容を分析し、コード修正を実施
  4. テスト・ビルド確認 → コミット
  5. **コミット後、自動で** `PATCH /api/bug-reports/{id}` を実行し、全対応済み報告のステータス（resolved）・対応内容・修正バージョンを更新する
  5. ユーザーに報告ではなく、管理画面（設定 → 不具合報告）で確認できる状態にする
- **確認ダイアログ回避ルール（全コマンド共通）:**
  - `$()` コマンド置換を使わない（値を事前計算して埋め込む）
  - `~` チルダ展開を使わない（フルパスで書く）
  - `\;` 等のバックスラッシュ+シェル演算子を使わない（xargs や専用ツールで代替）
  - パイプ+複数行スクリプトを使わない（コマンドを分けて実行）
  - `.claude/` 内にファイルを書き込まない（一時ファイルは `/tmp/` を使う）
- API呼び出し方法:
  ```
  # 取得（結果をファイルに保存してから処理）
  curl -s "https://karate.naocho.net/api/bug-reports" -H "Cookie: admin_auth=184deae3cb5b227cfcca890a13f9b0d0321b3b0adba290e3d69b5d8e945935d6" -o /tmp/bug-reports.json
  # → 別コマンドで python3 -c "..." で /tmp/bug-reports.json を読む

  # 更新
  curl -s "https://karate.naocho.net/api/bug-reports/{id}" -X PATCH -H "Cookie: admin_auth=184deae3cb5b227cfcca890a13f9b0d0321b3b0adba290e3d69b5d8e945935d6" -H "Content-Type: application/json" -d '{"status":"resolved","resolution":"対応内容","fixed_in_version":"コミットSHA先頭7桁"}'
  ```

## 全体レビュー
- 「全体レビュー」「全体レビューして」と指示されたら、以下の全観点でレビューを実行する:
  1. **docs/ 仕様書と実装の一致**: docs/ 内の各仕様書に記載された仕様が、実際のコードと一致しているか
  2. **仕様の矛盾・一貫性**: 仕様書内で矛盾した記述や一貫性のない定義がないか
  3. **仕様書間の整合性**: SPEC.md と docs/ 内の個別仕様書の間で齟齬がないか
  4. **テストカバレッジ**: 仕様に記載されているがテストが実装されていない箇所はないか
  5. **テスト期待結果と仕様の紐づき**: テストの期待値が仕様書の記述と整合しているか
  6. **その他必要な観点**: セキュリティ、エッジケース、型安全性、パフォーマンス等
- 複数の Explore エージェントを並列で使い効率的に調査する。

## 継続ルールの蓄積
- 「次からやって」「覚えといて」など継続的な指示は**このファイルに追記する**（Memory ファイルではなくここ）。
- 継続的に行った方がいいと思うものも判断して追記する。
- 迷う場合は「記憶しますか？」と確認してから追記する。
