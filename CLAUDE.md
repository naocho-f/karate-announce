# karate-announce 開発ルール

> **このファイルが唯一のルール正（Single Source of Truth）**
> 「覚えて」「次からやって」等の継続的指示は必ずこのファイルに追記する。
> Memory ファイルは補足情報（背景・外部リファレンス）のみ。ルール自体は置かない。
> この方針はユーザーと合意済み（2026-03-25）。理由：Memory ファイルはコンテキスト圧縮で
> 失われることがあり、CLAUDE.md だけが毎セッション確実に読み込まれるため。

---

## コミュニケーション
- 日本語は丁寧語（「〜しました」「〜です」）を使うこと。タメ口・体言止め禁止。
- 確認は最小限にする。テスト実行・DB変更・デプロイ等、このファイルに手順が書いてあることは確認不要で実行する。

## デプロイ
- `vercel --prod` は実行しないこと。デプロイは Stop hook が自動で行う。
- 実装完了後は `npm run build` でビルドエラーがないことを確認するだけでよい。
- SPEC.md を更新して応答を終えると Stop hook が自動でデプロイする。
- `git push` は不要。Stop hook が自動で行う。

## ビルド確認
- 実装後は `npm run build` でビルドエラーがないことを確認すること。

## テストコマンド
- `npm run dev`, `curl`, `npx tsc --noEmit` などテスト目的のコマンドは確認不要で実行すること。

## 実装後テスト（必須・最優先）
**実装完了と報告する前に、必ずテストを完了させること。テストなしの完了報告は禁止。**
`npm run dev` を起動し、`curl` や `npx tsc --noEmit` を使って実際に動作確認してからデプロイすること。これらのコマンドは確認不要で実行してよい。

実装が完了したら、デプロイ前に以下を必ずチェックすること。

### 1. データフロー検証（APIを変更・追加した場合）
以下の3点が一致しているか、コードを読んで確認する：
- フロントエンドが `fetch` で送っているフィールド名
- APIルートが `request.json()` で受け取っているフィールド名
- `insert` / `update` でDBに渡しているフィールド名

不一致があれば修正してからデプロイする。

### 2. エラーハンドリング確認（fetch を追加・変更した場合）
- `fetch` のレスポンスを `res.ok` でチェックしているか
- APIエラー時に UI が誤った状態に更新されないか（楽観的更新の場合は特に注意）
- エラー時にユーザーに通知しているか（alert または UI 表示）

### 3. ローディング状態確認（ボタン・フォームを追加・変更した場合）
- 処理中はボタンが `disabled` になっているか
- ローディング中に空のリストや「登録なし」テキストが表示されないか

### 4. 表示確認（データの保存・取得を変更した場合）
- 保存したデータが画面に正しく反映されるか、コードを追って確認する
- 画面を再ロードしたときに正しく表示されるか

### 5. マルチテナント化への配慮（DBテーブル・設定・認証を変更した場合）
- 団体固有のハードコード値（テキスト・ルール等）がないか → DB or config に外出しする
- 新しい外部サービス連携は環境変数で切り替え可能か
- Supabase 固有機能に依存していないか（PostgreSQL 標準の範囲で書く）
- 将来 `tenant_id` で分離できる構造になっているか

## DBスキーマ変更
- DBカラム追加・テーブル作成等が必要な場合、**ユーザーに依頼せず自分で** Supabase Management API 経由で実行する。確認も不要。
- 実行方法は memory/reference_supabase_sql.md を参照。
- 既存データのマイグレーションも同時に実行する。

## SPEC.md 更新（必須）
- 実装変更後は**指示されなくても自発的に** SPEC.md を更新する。毎回指示されるのを待たない。
- SPEC.md の更新はコード変更がすべて完了した後、コミット直前に1回だけ行う。
- 途中で SPEC.md を更新しない（後でコード変更すると Stop hook のフラグが再セットされる）。

## git コミット（必須）
**コミットは1つの実装に対して1回。分割コミット禁止。** 以下の手順を厳守：
1. コード変更を完了する
2. SPEC.md を更新する（先にやる。忘れるな）
3. `git add -A && git commit` を1回だけ実行する
- SPEC.md が diff に含まれていないコミットは禁止。コミット前に SPEC.md を変更したか必ず確認する。
- コミット後に「SPEC.md を忘れた」と気づいて追加コミットするのは手順違反。最初から含める。
- コミットメッセージは実装内容を日本語で簡潔に表すこと（例：「対戦表作成後に該当トーナメントへスクロール」）。
- 未コミットの変更が残っていると Stop hook がブロックして asyncRewake で再度呼ばれる。必ず commit まで完了させてから応答を終えること。

## ミス防止
- 同じミスを2回繰り返したら、3回目が起きないよう仕組み（Hook・スクリプト・設定）で対処する。
- 「次から気をつけます」だけで終わらせない。対策後は何をどう変えたか報告する。

## 継続ルールの蓄積
- 「次からやって」「覚えといて」など継続的な指示は**このファイルに追記する**（Memory ファイルではなくここ）。
- 継続的に行った方がいいと思うものも判断して追記する。
- 迷う場合は「記憶しますか？」と確認してから追記する。
